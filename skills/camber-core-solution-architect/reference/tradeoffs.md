# Tradeoffs — what Camber Core is bad at

This is the credibility instrument. A solution architect who only sells is not trusted.
Name the real limits during PROPOSE and FIT. Honesty here is what earns the call.

## Where Camber Core is the wrong answer

- **You want to cut headcount.** It amplifies a team's decision throughput. If the goal
  is fewer people doing the same work, the platform will frustrate you. Wrong product.
- **You want a tool you install once and forget.** The value is the *continuous fit* —
  keeping the deployment current as the business moves. Without the managed service it
  decays like any static system. If you want unmanaged software, buy unmanaged software.
- **Your business is unusually static.** Same people, tools, and workflows year over
  year? The continuous-fit value is muted. Consider Camber Core without the ongoing
  service.
- **You have a single tool and a single team.** If there's no entity sprawl, no
  cross-entity access need, and no institutional-knowledge capture problem, a focused
  point tool may serve you better and cheaper.

## Where engagements fail

- **No executive owner.** If nobody at the top will make decisions on what ships, the
  drafts-only model stalls — drafts pile up with no one to approve them.
- **Unwillingness to expose real workflows.** Skills are captured from how work
  *actually* gets done. If teams won't let the platform see real work, the skill library
  never accrues and the economics never compound.
- **Expecting instant magic.** Early deployment is mostly learning how the business
  works; the cost curve falls *over time*, not on day one. A buyer expecting week-one
  transformation will be disappointed.

## Honest worst case

The first months are investment: research, integration, and pattern-finding before the
per-task cost starts falling. If the relationship ends early, the client keeps a working
platform but may not have reached the compounding-returns phase. That's a real risk, and
naming it is the point.
